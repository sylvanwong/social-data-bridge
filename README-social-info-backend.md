# social_info 后端改造说明

本文档对应前端 `VideoDataFetch.vue` 的新请求格式，用于把飞书单元格原值透传给后端，由后端统一解析作品链接。

## 改造目标

- 前端不再负责平台识别、短链解析、链接合法性判断。
- 后端统一接收飞书单元格原值并解析出最终作品链接。
- 兼容旧前端仅传 `{ url }` 的请求。

## 新请求格式

前端请求接口仍然是：

- `POST /social/api/v1/feishu/social_info`

请求体新增字段：

```json
{
  "url": "可选，前端提取到的首个字符串链接",
  "raw_value": "可选，飞书单元格原始值",
  "source": "feishu_bitable_video_fetch"
}
```

说明：

- `url`
  - 兼容旧逻辑的兜底字段。
  - 文本字段时，通常就是单元格文本 `trim()` 后的内容。
  - URL 字段时，通常是前端从 URL 结构数组里取到的第一个 `item.link`。
- `raw_value`
  - 飞书单元格原始值，可能是字符串，也可能是 URL 字段数组。
  - 后端应优先解析这个字段。
- `source`
  - 调用来源标识，当前固定为 `feishu_bitable_video_fetch`。
  - 仅用于日志和兼容排查，不参与核心业务判断。

## 兼容要求

后端必须同时兼容两种入参：

1. 旧格式

```json
{
  "url": "https://..."
}
```

2. 新格式

```json
{
  "url": "https://...",
  "raw_value": [
    {
      "type": "url",
      "text": "示例链接",
      "link": "https://..."
    }
  ],
  "source": "feishu_bitable_video_fetch"
}
```

## 建议解析顺序

建议把解析逻辑封装成公共方法，例如 `resolve_social_url(payload)`。

处理顺序：

1. 读取 `raw_value`
2. 如果 `raw_value` 是非空字符串
   - 直接作为原始输入文本
   - 从文本中提取首个候选链接
3. 如果 `raw_value` 是数组
   - 遍历数组项
   - 优先提取 `type === "url"` 且存在 `link` 的项
   - 如有必要，也可兜底提取其中可见文本再做 URL 抽取
4. 如果 `raw_value` 未解析出候选值
   - 回退使用 `url`
5. 对候选值做统一规范化
   - 去掉前后空格
   - 从分享文案中抽取 URL
   - 展开短链
   - 去掉无关 query 参数
   - 识别平台
   - 识别链接类型
6. 校验是否为支持的平台作品链接
7. 把解析后的标准链接传入现有 `social_info` 抓取逻辑

## 平台和链接类型判断建议

当前前端页面只用于音视频作品获取，后端建议至少区分：

- 支持的平台
  - 抖音
  - 小红书
- 支持的链接类型
  - 作品详情链接
- 不支持的链接类型
  - 主页链接
  - 合集链接
  - 搜索页链接
  - 未识别平台链接

如果当前后端已有更完整的平台路由，直接复用即可，但这里的校验建议统一放在解析层做。

## 返回约定

建议保持当前响应结构不变：

成功：

```json
{
  "sta": 0,
  "data": {}
}
```

失败：

```json
{
  "sta": 1,
  "msg": "错误信息"
}
```

错误信息建议尽量明确，便于前端直接回写到表格：

- `未识别到可用链接`
- `链接解析失败`
- `暂不支持该平台链接`
- `当前链接不是作品链接`
- `作品信息获取失败`

## 日志建议

建议在后端记录以下信息，便于灰度排查：

- `source`
- 原始 `url`
- `raw_value` 的类型
- 最终解析出的标准链接
- 平台识别结果
- 失败原因

注意不要记录敏感凭证信息。

## 推荐上线顺序

1. 后端先上线，兼容新旧两种请求体
2. 前端再发版，开始透传 `raw_value`
3. 观察实际日志样本，补齐解析规则

## 当前前端已做的变更

`VideoDataFetch.vue` 已按新协议发送：

```json
{
  "url": "...",
  "raw_value": "...或数组...",
  "source": "feishu_bitable_video_fetch"
}
```

并修复了过滤空值后可能写错记录行的问题：前端现在按 `{ recordId, url, rawValue }` 成对处理，不再依赖过滤后的索引位置。
